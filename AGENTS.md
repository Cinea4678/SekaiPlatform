# AGENTS.md

## 文档

- `docs/design/` 目录用于存储项目设计文档。
- `docs/plan/` 目录用于存储项目实施计划。
- 涉及架构、数据模型、接口或关键业务流程的改动时，先查看 `docs/design/index.md` 及相关设计文档。

## 代码注释

- 新增或修改公共 API、跨模块调用入口、公开/内部类型、record、接口、扩展方法、endpoint 映射方法、后台任务入口、数据库实体/上下文、配置选项、关键业务函数和关键测试辅助节点时，应补充简洁准确的 XML 文档注释。
- 注释应说明该成员在业务或架构中的职责、关键参数含义和返回值语义，避免只复述代码实现。
- 生成代码、迁移文件和显而易见的局部实现细节不必为了覆盖率强行补注释。

## 模块组织

- 鼓励 Agent 按领域和职责优雅地组织目录与模块，优先让相关类型、endpoint、契约、服务、builder、helper 分布在清晰的子目录/子模块中。
- 当某个目录或模块下 `.cs` 文件数量明显增多，或单个文件承担多个职责时，应考虑拆分为更具体的子目录、子模块或协作类型。
- 拆分时保持行为不变，遵循现有命名空间、项目结构和依赖方向，避免为了形式整洁引入过度抽象。
